# 작업 보고서: 전투결과 판정 구현

**날짜**: 2026-03-05
**작업자**: Claude

## 요청 내용

마지막 몬스터 또는 마지막 Pawn 사망 시 `BattleResultPopup`을 호출하는 전투 결과 판정 로직을 구현해달라는 요청.
승패 판정의 아키텍처 위치(TurnManager vs StageManager)에 대한 설계 논의 후 이벤트 기반 구조로 최종 구현.

## 수행한 작업

- 코드베이스를 분석하여 기존 사망 처리 경로(TurnManager, StagePage) 및 BattleResultPopup API 파악
- 승패 판정 책임 위치에 대해 TurnManager vs StageManager 구조적 장단점 분석
- StageManager에 `onBattleEnd` 이벤트 및 `CheckBattleResult()` 메서드 추가
- TurnManager.ExecuteAttack() — 사망 애니메이션 완료 후 `CheckBattleResult()` 호출 연결
- StagePage.UseCard() — 카드 데미지 경로의 사망 콜백에 `CheckBattleResult()` 호출 연결
- StageManager가 UIManager를 직접 호출하는 레이어 위반 문제를 지적받아 이벤트 기반으로 리팩토링
- StagePage에서 `onBattleEnd` 이벤트 구독/해제 및 `OnBattleEnd()` 핸들러 구현

## 수정된 파일

| 파일 경로 | 변경 유형 | 변경 내용 요약 |
|-----------|-----------|----------------|

| `Assets/Scripts/Logic/Manager/TurnManager.cs` | 수정 | `ExecuteAttack()` 사망 처리 후 `StageManager.Instance.CheckBattleResult()` 호출 추가 |

## 주요 결정 사항

- **승패 판정은 StageManager 담당**: 유저/적 액터 목록(`_dicUserActor`, `_dicEnemyActor`)을 직접 소유하므로 생존 여부 판단 권한이 있음. TurnManager는 턴 제어가 책임이므로 범위 초과.
- **이벤트 기반 구조 채택**: StageManager → UIManager 직접 호출은 Manager 레이어가 UI 레이어를 참조하는 레이어 위반. `onBattleEnd` 이벤트를 발행하고 StagePage(UI 레이어)가 구독하여 팝업 호출하는 방식으로 레이어 분리.
- **사망 애니메이션 완료 후 판정**: `Die()` 콜백 이후에 `CheckBattleResult()`를 호출하여 애니메이션이 끝난 뒤 팝업이 열리도록 처리.

## 참고 사항

- 사망 경로가 두 곳(몬스터 공격: TurnManager / 카드 사용: StagePage)이므로 양쪽 모두 `CheckBattleResult()` 호출이 연결되어 있음.
- `allEnemiesDead`와 `allUsersDead`가 동시에 true인 경우(동귀어진) 승리로 처리됨. 필요 시 별도 정책 논의 필요.
